# 기여 가이드

SKALA Review Skill에 관심을 가져주셔서 감사합니다. 처음 오픈소스에 기여하는 분도 아래 순서대로 진행하면 됩니다.

## 기여할 수 있는 내용

- 비전공자가 이해하기 어려운 안내 문구 개선
- 복습 질문 흐름과 플랫폼별 출력 형식 개선
- 새로운 복습 노트 템플릿 제안
- Codex·Claude Code Skill 또는 ChatGPT·Gemini·Claude 웹 프롬프트 개선
- 설치·실행 오류 수정과 테스트 보강
- 공개된 범위의 교육 과정 연결 정보 개선

## 포함하면 안 되는 내용

- 배포가 제한된 강의 PDF, 슬라이드 또는 교재 원문
- 교수자나 교육생의 개인정보 및 수업 중 비공개 발언
- 개인 복습 노트와 실제 대화 기록
- API 키, 토큰, 계정 정보 또는 로컬 절대 경로
- 출처나 사용 허가를 확인할 수 없는 이미지·문서

예시가 필요하다면 실제 수업 자료 대신 직접 만든 짧은 가상 예시를 사용해 주세요.

## 기여 절차

1. 작업 전 [Issue](https://github.com/yongchane/skala-review-skill/issues)를 검색해 같은 제안이나 오류가 있는지 확인합니다.
2. 저장소를 Fork한 뒤 작업 브랜치를 만듭니다.

   ```bash
   git checkout -b docs/쉬운-설명-개선
   ```

3. 공통 참고자료·템플릿·스크립트는 `core/`에서 수정합니다. Codex 지침과 메타데이터는 `platforms/codex/skala-review/`, Claude Code 지침은 `platforms/claude-code/skala-review/SKILL.md`에서 수정합니다. 패키징으로 생성되는 `references/`, `templates/`, `scripts/` 사본은 직접 수정하지 않습니다. 웹 AI용 프롬프트는 `web-prompts/`에서 직접 수정합니다.
4. 배포본을 동기화하고 검증합니다.

   ```bash
   node tools/package-skill.mjs
   node tools/validate.mjs
   ```

5. 변경 이유가 드러나는 커밋을 만들고 Pull Request를 엽니다.

   ```bash
   git commit -m "docs: 복습 시작 안내를 쉽게 수정"
   ```

## 버전과 Release

- 일반 기능 변경은 Pull Request로 `main`에 반영합니다.
- Release를 준비할 때는 루트 `VERSION`과 `plugins/skala-review/.codex-plugin/plugin.json`의 버전을 같은 SemVer 값으로 올립니다.
- `vX.Y.Z` 태그는 관리자만 검증 완료 후 생성합니다.
- 태그가 푸시되면 GitHub Actions가 다시 검증하고 플랫폼별 압축 파일과 GitHub Release를 생성합니다.
- 개인 강의 자료, 복습 기록, 계정 정보가 Release 파일에 포함되지 않았는지 확인합니다.

## 문서 작성 기준

- 비전공자도 한 번에 이해할 수 있는 쉬운 한국어를 우선합니다.
- 한 문장에 너무 많은 개념을 넣지 않습니다.
- 전문용어가 필요하면 처음 나올 때 짧게 풀이합니다.
- 사용자를 채점하거나 나무라는 표현을 사용하지 않습니다.
- 원본 수업 자료를 추측하거나 재현하지 않습니다.
- 기술 식별자, 명령어, YAML 필드명은 호환성을 위해 원문을 유지합니다.

## 웹 프롬프트 기여 방법

웹 프롬프트는 Skill 설치가 어려운 교육생이 복사해 사용하는 공개 자료입니다.

1. `web-prompts/skala-review-prompt.md` 또는 `web-prompts/skala-quick-draft-prompt.md`를 선택합니다.
2. 파일의 개요·사용 방법과 `복사할 프롬프트`가 서로 일치하도록 수정합니다.
3. 실제 강의 원문 대신 직접 만든 짧은 가상 입력으로 시험합니다.
4. ChatGPT·Gemini·Claude 중 확인한 환경과 결과를 Pull Request에 적습니다.

프롬프트를 수정할 때는 다음 항목을 확인합니다.

- 첫 실행 안내가 한 번만 표시되는가?
- 사용 가능한 기능과 요청 예시가 짧고 이해하기 쉬운가?
- 질문식 복습은 질문을 한 번에 하나만 하는가?
- 복습 글 초안과 플랫폼별 작성 가이드가 지나치게 길지 않은가?
- 제공되지 않은 수치·출처·수업 내용을 만들지 않는가?
- 자동 게시 또는 다른 대화의 파일 기억을 약속하지 않는가?

## Pull Request 확인 목록

- 변경 목적과 사용자에게 주는 효과를 설명했는가?
- `core/` 변경 후 배포본을 다시 패키징했는가?
- 웹 프롬프트 변경 시 개요·사용 방법·실행 지침을 함께 확인했는가?
- `node tools/validate.mjs`가 통과했는가?
- 강의 원문, 개인정보, 비밀 값이 포함되지 않았는가?
- README 또는 사용 방법 변경이 필요하면 함께 수정했는가?

## Pull Request 작성 방법

변경이 끝나면 원본 저장소의 `main` 브랜치를 대상으로 Pull Request를 만듭니다. PR을 열면 작성 양식이 자동으로 표시됩니다. 처음 PR을 작성한다면 모든 문장을 완벽하게 쓰지 않아도 괜찮습니다. 확인한 범위에서 작성해 주세요.

### 변경 내용

무엇을 바꿨는지 1~3줄로 작성합니다.

```text
- Gemini 웹에서 복습 지침을 등록하는 방법을 추가했습니다.
- 웹에서 저장한 Markdown을 다시 복습하는 흐름을 보완했습니다.
```

### 변경 이유

어떤 사용자의 문제를 해결하는지 설명합니다.

```text
터미널을 사용하기 어려운 교육생도 웹 브라우저에서 같은 복습 흐름을 사용할 수 있게 하기 위해 수정했습니다.
```

### 확인 방법

실행한 명령과 직접 확인한 사용 시나리오를 적습니다.

```text
- node tools/package-skill.mjs
- node tools/validate.mjs
- 웹 프롬프트를 새 대화에 붙여 넣고 첫 안내와 기능 선택이 정상인지 확인
```

### 확인 목록

PR 양식의 체크 항목을 읽고 실제로 확인한 항목만 선택합니다. 검증하지 못한 항목은 선택하지 않고 PR 설명에 이유를 적어 주세요.

작은 오탈자 수정도 환영합니다. 처음 기여한다면 Issue에 먼저 질문해도 괜찮습니다.
